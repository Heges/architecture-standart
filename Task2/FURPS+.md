# Task 2. FURPS+ таблица требований

| Код | Требования | Комментарий |
|-----|------------|-------------|
| F | Клиент должен видеть на сайте список доступных депозитных продуктов с актуальными ставками. | Сайт становится каналом привлечения новых клиентов. Для этого ставки должны быть доступны сайту из системного источника, а не из локального Excel-файла. |
| F | Новый клиент должен иметь возможность подать заявку на депозит на сайте, указав ФИО и номер телефона. | С сайта передаются персональные данные, поэтому дальше нужно учитывать защиту трафика. |
| F | Заявка с сайта должна попадать менеджеру кол-центра для последующего звонка клиенту. | Новый клиент не открывает депозит полностью через сайт. После звонка ему нужно прийти в отделение для идентификации. |
| F | Действующий клиент должен видеть в интернет-банке доступные депозитные продукты, актуальные ставки и персональные условия. | Это основной сценарий для клиентов, которые уже обслуживаются в банке. |
| F | Действующий клиент должен иметь возможность выбрать счет списания, сумму депозита и подать заявку в интернет-банке. | Это главный сценарий MVP для интернет-банка. |
| F | Подача заявки в интернет-банке должна подтверждаться кодом из СМС. | Требование по безопасности операции. |
| F | Сайт и интернет-банк должны создавать заявку на депозит и передавать ее в обработку бэк-офиса. | В MVP бэк-офис еще остается в процессе, автоматическое открытие депозита относится к следующему этапу. |
| F | Менеджер бэк-офиса должен обработать заявку, подтвердить ставку и инициировать открытие депозита в АБС. | АБС остается основной учетной системой банка. |
| F | Клиент должен получить СМС после подтверждения ставки и после открытия депозита. | Это требование по информированию клиента о результате обработки. |
| F | Ставки должны вестись в контролируемом системном процессе, а не только в локальных Excel-файлах. | Иначе сайт и интернет-банк не смогут надежно показывать актуальные ставки. |
| F | Доступ сотрудников к заявкам и ставкам должен быть разграничен по ролям. | Депозитный и кредитный бэк-офисы не должны видеть лишние данные друг друга. |
| F | По заявке должен сохраняться статус обработки. | Без статуса клиент и сотрудники не будут понимать, где сейчас находится заявка. |
| U | Интерфейс сайта и интернет-банка должен быть понятным клиенту и соответствовать дизайн-системе банка. | В задании отдельно указаны привычные цвета и брендированные элементы. |
| U | Клиент должен получать понятную обратную связь после отправки заявки и после ввода СМС-кода. | Это снижает риск повторных отправок и обращений в поддержку. |
| U | На сайте должно быть понятно, что новому клиенту после заявки нужен звонок менеджера и личное посещение отделения. | Это ограничение процесса идентификации новых клиентов. |
| R | Сервисы, участвующие в подаче и обработке заявок, должны работать 24/7 с целевой доступностью 99,9%. | Требование явно указано в задании. Для MVP его можно достигать поэтапно, так как текущий интернет-банк к этому не готов. |
| R | Сайт и интернет-банк не должны напрямую создавать критическую нагрузку на АБС. | База АБС уже перегружена, а сама АБС масштабируется в основном вертикально. |
| R | При временной недоступности АБС заявка клиента не должна теряться. | Для MVP допустима отложенная обработка, но потеря заявки неприемлема. |
| R | Система должна корректно обрабатывать сбои отправки СМС. | СМС используется и для подтверждения операции, и для уведомлений. |
| R | По ключевым интеграциям должны быть логи, мониторинг и понятные ошибки. | Без этого сложно поддерживать процесс 24/7. |
| P | Отклик пользовательских операций на сайте и в интернет-банке должен быть быстрым. | В задании сказано, что сейчас есть проблемы с производительностью и некоторые справочные данные загружаются больше секунды. |
| P | Сайт, интернет-банк и сервис депозитных заявок должны масштабироваться отдельно от АБС. | Нельзя строить решение так, чтобы рост заявок с сайта и интернет-банка решался только увеличением нагрузки на Oracle БД АБС. |
| S | При доработках нужно максимально использовать технологии, которые уже есть в банке. | Это снижает стоимость разработки и поддержки. |
| S | Новая функциональность интернет-банка должна минимизировать изменения ядра платформы подрядчика. | Ядро интернет-банка зависит от подрядчика, поэтому лучше вынести новую бизнес-логику отдельно. |
| S | Решение должно быть документировано и расширяемо для следующих продуктов, включая кредиты. | В задании сказано, что позже банк планирует подачу заявок на кредиты через сайт и интернет-банк. |
| +R | Персональные и чувствительные данные должны передаваться по защищенному каналу. | Это явно указано для сайта и интернет-банка. |
| +R | Новый клиент не может полностью открыть депозит через сайт без очной идентификации. | После заявки с сайта клиент должен прийти в отделение. |
| +R | Интернет-банк в текущей версии не совместим с Kafka. | Kafka можно рассматривать как перспективу, но не как обязательное решение для MVP. |
| +R | Нужно избегать прямой работы интернет-банка с API или базой данных АБС в новом процессе. | Это следует из перегруженности АБС и требований доступности. |
